Данная практическая работа является **архитектурным фундаментом** моего курсового проекта. Реализованный прототип полностью перенесен на бизнес-логику Консалтингового центра (КЦ):

*   **RBAC (Role-Based Access Control)**: Разграничение прав доступа между Консультантом ЦК и Инженером ИЦ.

## 📂 2. Структура проекта (Декомпозиция)
Проект разбит на независимые компоненты для соблюдения принципа **Single Responsibility**:
*   `src/components/Layout.jsx`: Каркас приложения (обертка) с использованием `children` и мониторингом BOM-событий.
*   `src/components/AddTaskForm.jsx`: Управляемая форма для регистрации инженерных заявок.
*   `src/components/FilterPanel.jsx`: Панель фильтрации по жизненному циклу (Новая, В расчете, Завершено).
*   `src/components/TaskList.jsx`: Компонент-контейнер для безопасного рендеринга реестра.
*   `src/components/TaskCard.jsx`: Презентационный компонент строки реестра с логикой расчета стоимости.
*   `src/components/RoleToolbar.jsx`: Панель управления правами доступа (RBAC).
*   `App.js`: Единственный источник истины (Single Source of Truth), логика `localStorage` и бизнес-логика фильтрации.

## 🎤 3. Interview Corner (Ответы на вопросы)

### 1. Что такое «однонаправленный поток данных» в React? Почему это удобно?
Данные всегда передаются строго от Родителя к Потоку через `props`. Это исключает ситуацию, когда компоненты хаотично меняют данные друг друга. Это удобно, так как делает отладку прозрачной: если в таблице ошибка, мы ищем её только в источнике данных (`App.js`).

### 2. Что такое lifting state up? Приведите пример из работы.
Это «подъем состояния» к общему предку. В моей работе форма добавления и список заявок живут в разных компонентах, но зависят от одних данных. Я поднял состояние `tasks` в `App.js`, чтобы все компоненты работали с актуальной базой одновременно.

### 3. Как дочерний компонент сообщает родителю о событии? Почему нельзя изменить state родителя напрямую?
Дочерний компонент вызывает **callback-функцию**, переданную ему родителем через `props` (например, `onStatusChange`). Напрямую менять state родителя нельзя, так как это нарушит принцип иммутабельности: React не сможет сравнить старую и новую версии данных и не обновит интерфейс.

### 4. Что такое управляемый компонент (controlled component)? Зачем нужен атрибут value у input?
Это элемент формы, значение которого полностью контролируется состоянием (`state`) React. Атрибут `value` связывает отображаемый текст с переменной в памяти. Это необходимо для мгновенной валидации данных и реализации «живого» поиска по реестру.

### 5. Что такое паттерн «компонент-обёртка» (wrapper component)? Как работает children?
Это компонент (у меня `Layout.jsx`), который создает общую визуальную рамку. Свойство `children` — это механизм композиции, позволяющий вкладывать любой контент внутрь обертки, избавляя от дублирования кода шапки и подвала.

### 6. Почему в React предпочтительна композиция, а не наследование?
Композиция позволяет собирать интерфейс из мелких деталей как конструктор LEGO. Наследование создает жесткие иерархии, которые крайне сложно менять. В React проще вложить один компонент в другой, чем пытаться выстроить их «родственные» связи.

### 7. Зачем нужен useEffect для работы с localStorage? Что такое «побочный эффект»?
Работа с `localStorage` (диском) — это действие вне контроля React, т.е. «побочный эффект». `useEffect` позволяет синхронизировать состояние приложения с памятью браузера строго после того, как интерфейс был успешно отрисован.

## ❓ 4. Ответы на «Вопросы на засыпку»

**1. Проблема использования Date.now() для ID:**
Использование `Date.now()` создает риск коллизии: если два инженера нажмут кнопку «Создать» в одну и ту же миллисекунду, у них будут одинаковые ID. В React это приведет к ошибкам рендеринга (duplicate keys) и некорректному удалению объектов.  
*Решение для PRO:* Использовать `crypto.randomUUID()` или комбинированный ключ (время + случайное число).

**2. Почему передача role в TaskCard — это не совсем "чисто"?**
Передача `role` внутрь каждой карточки создает лишнюю зависимость.  
*Более чистый подход:* Проверять роль на уровне `TaskList`. Если у пользователя нет прав на удаление, мы просто не передаем функцию `onDelete` в пропсы `TaskCard`. Если пропс `onDelete` равен `null`, карточка сама скроет кнопку. Это делает компонент `TaskCard` более универсальным.
